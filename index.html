01 — The Journey

From raw data to a reasoning system

Every step below was a concept learned in the course and then applied for real on the IBM HR dataset.

  1. Understanding the data

    Loaded 1,470 employee records, dropped non-informative columns, handled the target, and confirmed a clean dataset with a meaningful class imbalance (~16% leavers) — the single fact that shaped every decision afterwards.

  2. Exploratory Data Analysis

    Twelve visual stories — overtime, income, age, job role, marital status, satisfaction, tenure — to find the real drivers of attrition before modelling anything.

  3. Preprocessing — scaling & normalizing

    Learned why feature scaling matters: standardizing features let Logistic Regression converge in 18 iterations instead of failing after 1,000, and made its coefficients interpretable. Compared standardization vs min-max vs row-normalization and chose the right tool.

  4. Modelling — from one tree to ensembles

    Logistic Regression → Decision Tree → Random Forest, then the ensemble family: Bagging (parallel, variance↓), Boosting (sequential, bias↓), and Stacking (a learned blend of all three).

  5. Evaluation — beyond accuracy

    Confusion matrices, precision/recall, F1, ROC-AUC, and cross-validation. Learned that on imbalanced data accuracy lies — and that the threshold you classify at is a design decision, not a default.

  6. Interpretation & action

    Turned coefficients into a human-readable "why", and a 13-rule engine into personalised retention recommendations. A model is only useful when someone can act on it.

03 — Data & Methodology

Where the data came from & how it was prepared

Good models start with honest data work. Here's exactly how the dataset was sourced, cleaned, and made model-ready.

📊 The dataset

The project uses the IBM HR Analytics Employee Attrition & Performance dataset — a well-known, openly available dataset published by IBM Watson Analytics and widely distributed via Kaggle. It is a fictional dataset created by IBM data scientists to study the factors behind employee attrition, which makes it ideal for learning without privacy concerns.

  • Rows1,470 employees
  • Original columns35 attributes
  • TargetAttrition — Yes / No
  • Class balance~16% left · ~84% stayed
  • Missing valuesNone (clean dataset)
  • FileWA_Fn-UseC_-HR-Employee-Attrition.csv

🎯 The target

Each employee is labelled Attrition = Yes (left the company) or No (stayed). This was mapped to 1 / 0 so the models could learn a binary classification: will this person leave?

⚖️ The imbalance challenge

Only about 1 in 6 employees actually left. This imbalance shaped every later choice — from balanced class weights to a tuned decision threshold — because a naïve model could score 84% accuracy by simply predicting "nobody leaves".

The preprocessing pipeline, step by step

  1. 1

    Load & inspect

    Read the CSV into a pandas DataFrame (1,470 × 35), checked shape, data types, and confirmed there were zero missing values — so no imputation was needed.

  2. 2

    Drop non-informative columns

    Removed columns that carry no predictive signal: EmployeeNumber, EmployeeCount, Over18, and StandardHours (all constant or just identifiers), plus low-signal rate fields (MonthlyRate, DailyRate, HourlyRate, PerformanceRating) — keeping 26 meaningful features.

  3. 3

    Encode the target

    Mapped Attrition from "Yes"/"No" to 1/0 so it could be used as a numeric classification label.

  4. 4

    Encode categorical features

    Categorical columns (e.g. Department, JobRole, MaritalStatus, OverTime) were converted to numbers with Label Encoding, and the fitted encoders were saved so any new employee can be encoded identically at prediction time.

  5. 5

    Train / test split

    Split 80% train / 20% test using stratified sampling, which preserves the ~16% attrition ratio in both sets so the test score reflects real-world balance.

  6. 6

    Feature scaling

    Applied StandardScaler (mean 0, std 1), fitted only on the training set to avoid leakage. This was essential for Logistic Regression — it cut training from a non-converging 1,000 iterations down to 18 and made coefficients comparable. Tree-based models were fed the raw, unscaled features since they're scale-invariant.

  7. 7

    Statistics for recommendations

    Computed per-feature averages for employees who stayed vs left. These power the system's explanation layer — e.g. "this salary is $2,000 below the average of retained employees."

The metrics, in plain English

Accuracy

Share of all predictions that were correct — but misleading when 84% never leave.

Precision

Of those we flagged as "will leave", how many actually did. High precision = few false alarms.

Recall

Of everyone who actually left, how many we caught. The metric that matters most here.

F1-Score

The balance between precision and recall — one number for the trade-off.

ROC-AUC

How well the model ranks risk regardless of threshold. 0.5 = guessing, 1.0 = perfect.

Confusion Matrix

The full breakdown of true/false positives and negatives — which errors, not just how many.

Risk factors don't act alone — they multiply. Low income + overtime employees reach roughly 53% attrition, far above any single factor.
